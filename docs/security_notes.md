# セキュリティ注意事項

> **このアプリはlocalStorageを使用したプロトタイプです。本番運用にはバックエンド認証・DB管理が必要です。**

---

## このアプリのセキュリティ前提

**本アプリは社内プロトタイプ用途を前提として作成されています。**  
本番環境・外部公開・機密情報の管理には使用しないでください。

---

## 既知のセキュリティ上の制限

### 1. クライアントサイドのみの認証

HTML / CSS / JavaScript のみで実装された認証は、本来の意味での「認証」ではありません。

- ソースコードを見れば認証ロジックが丸見えになる
- ブラウザの開発者ツールから localStorageを直接操作することでアクセス制御を回避できる
- ネットワーク通信を伴う認証ではないため、セッションの安全な管理ができない

### 2. パスワードの平文保存

本アプリではパスワードを localStorage にプレーンテキストで保存しています。

- ハッシュ化・ソルト処理は行っていない
- 同じ端末の他のユーザーがブラウザの開発者ツールでパスワードを確認できる
- XSSが成立した場合、パスワードを含む全データが漏洩する

### 3. XSS（クロスサイトスクリプティング）への対応

- innerHTML を使って動的にHTMLを描画する箇所は、必ずユーザー入力をエスケープすること
- textContent / innerText を優先して使用すること
- 実装時に必ず確認が必要

### 4. データはすべてブラウザローカルに存在する

- localStorage はそのブラウザ・端末に紐づくため、別の端末からはアクセスできない
- ブラウザのデータ削除・キャッシュクリアでデータが消える
- 複数人での同時利用・データ共有には対応していない

---

## 本番利用時に必要な対応

本アプリを本番環境で使用する場合は、以下の対応が必須です。

1. **バックエンド認証の導入**  
   サーバーサイドで認証・認可を管理する。JWTやセッションCookieを使用する。

2. **パスワードのハッシュ化**  
   bcrypt などのアルゴリズムでパスワードをハッシュ化して保存する。

3. **HTTPS通信の強制**  
   通信を暗号化し、中間者攻撃を防ぐ。

4. **SQLインジェクション対策**  
   DBを使用する場合はプリペアドステートメントを使用する。

5. **XSS対策**  
   入力値のサニタイズ・エスケープを徹底する。

6. **CSRF対策**  
   フォーム送信にCSRFトークンを付与する。

## 将来的なバックエンド移行候補

本アプリはバックエンド移行を考慮した設計にしており、localStorage操作は `storage.js` に集約しています。  
移行時は `storage.js` の各関数をAPIコール（fetch）に置き換えることで対応できます。

### Supabase Auth
- メールアドレス / OAuth / マジックリンク認証
- Row Level Security (RLS) でテーブル単位のアクセス制御が可能
- PostgreSQL ベースで本番利用に適している

### Firebase Authentication
- メールアドレス / OAuth / 電話番号認証
- Firestore / Realtime Database と組み合わせて使える
- Googleアカウントでのログインが容易

### Google Apps Script (GAS) + Googleアカウント認証
- Googleアカウントで認証し、スプレッドシートをDBとして使う
- 社内G Suite / Google Workspace 環境であれば導入コストが低い
- 大規模利用には不向き

### PHP + DB (MySQL / PostgreSQL)
- 既存のレンタルサーバー環境で動作しやすい
- セッション管理・パスワードハッシュ化など標準的な実装が可能

### Node.js + DB
- Express / Fastify などのフレームワークと組み合わせる
- TypeScript での型安全な開発が可能
- フロントエンドと言語を統一できる

## 開発時の注意事項

- 本番の従業員データや機密情報を開発環境に入力しないこと
- テストデータには架空の人物情報を使用すること
