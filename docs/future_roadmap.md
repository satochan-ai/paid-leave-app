# 今後の拡張予定（ロードマップ）

現在のプロトタイプをベースに、本番利用に向けた拡張優先度を整理します。

---

## 優先度：高（本番化に必須）

| 項目 | 概要 |
|------|------|
| バックエンド移行（Supabase / Firebase） | localStorageからサーバーDBへ移行し、複数端末共有・データ永続化を実現する |
| 本番認証の導入 | JWTやセッションCookieを使ったサーバーサイド認証に切り替える |
| パスワードハッシュ化 | bcrypt等でパスワードをハッシュ化して保存する |
| 初回ログイン時パスワード変更 | 管理者が設定した仮パスワードを、初回ログイン時に変更させる |
| HTTPS通信の強制 | 本番環境での通信を暗号化する |

---

## 優先度：中（業務効率化）

| 項目 | 概要 |
|------|------|
| 有給申請承認フロー | 社員が申請→管理者が承認/却下するワークフローを追加する |
| メール通知 | 失効予定アラート・承認結果をメールで通知する |
| CSVインポート | CSVから社員データ・有給履歴を一括取り込みする |
| 有給取得予定管理 | 将来の有給取得予定を登録・カレンダー表示する |
| 管理者による複数承認者設定 | 承認権限を持つ担当者を複数設定できるようにする |

---

## 優先度：低（付加価値）

| 項目 | 概要 |
|------|------|
| PDF出力 | 有給取得履歴・サマリーをPDFで出力する |
| Excel（xlsx）出力 | CSV以外にxlsx形式でも出力できるようにする |
| グラフ表示 | 有給取得率・残日数推移などをグラフで可視化する |
| 休職期間を考慮した厳密計算 | 休職期間を出勤率・付与日数の計算に反映する |
| 出勤率8割判定 | 労働基準法上の出勤率要件を判定し、付与対象外を検出する |
| 時間単位有給 | 時間単位での有給取得に対応する |
| 自動バックアップ | 定期的にlocalStorageデータを自動バックアップする |
| 勤怠システム連携 | 既存の勤怠システムとAPIで連携する |

---

## バックエンド移行候補

### Supabase
- PostgreSQL + RLS（行レベルセキュリティ）でアクセス制御
- Auth機能でメール認証・OAuth対応
- `storage.js` の関数をAPIコールに差し替えるだけで移行可能

### Firebase
- Firestore + Authentication
- Googleアカウント連携が容易
- リアルタイム同期が必要な場合に有利

### Google Apps Script + Googleスプレッドシート
- Google Workspaceを使っている組織なら最低コストで移行可能
- Googleアカウント認証で社員管理が容易

### Node.js + MySQL / PostgreSQL
- 既存サーバーがある環境で使いやすい
- TypeScriptで型安全な実装が可能

---

## 移行設計メモ

本アプリは `storage.js` にlocalStorage操作を集約しており、移行コストを最小化する設計になっています。

```
getUsers() → fetch('/api/users')
saveUsers() → fetch('/api/users', { method: 'PUT', ... })
```

各関数をAPIコールに差し替えることで、アプリロジックを変更せずにバックエンド移行できます。
